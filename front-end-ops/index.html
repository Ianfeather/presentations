Front End Ops

This isn't about Grunt, Bower, Yeoman…

It's a collection of concepts, rather than tools.

I'd never heard the term ‘Front End Ops’ before the Alex Sexton article. To me it was still DevOps but it was saying, "Step up, this is no longer just the work of back end devs." As it shouldn't be, this stuff is fun.

It was talking to Oliver Ash that made me decide to do this talk. We discussed this Catch 22 situation around a lot of these topics: where it was very hard to actually learn them without being exposed to them (usually in large organisations) - and equally difficuly to get the role where you would be exposed to them, without the skills.

Key Concepts

    “The metric by which a front-end operations engineer would be judged is speed.” - Alex Sexton, Smashing Magazine

    Your job (as a developer) is to:

  • Get the best code up to the server as fast as possible.
  • Begin the User Experience as fast as possible.
  • Enable fast feedback.

    By finding ways to:

  • Offload as much as possibe
  • Tighten the feedback loop

Author code

Focus on offloading risk away from the developer.

  • Create gateposts to protect the production environment.
  • Ensure these gateposts work for the developer.
  • You want to be notified of a bug as early as possible.

We need to manage the relationship between speed & risk:

  • Changing code on the server - fast but risky-as-hell.
  • Synthetic performance testing - slow but safe.

Commit

Our first Gatepost.

An opportunity to run small, fast, composable tasks:

  • Remove console.logs, debuggers, local gem refs etc.
  • Styleguide compliance.
  • Code coverage, complexity.

Push

Our second Gatepost.

An opportunity to run slightly longer tasks.

Errors discovered here are already committed.

Example: Visual Regression tooling.

Merge

Our third Gatepost.

Ideally this is as simple as a Pull Request and Code Review.

This should be fast, because Code Review should have been happening organically throughout the feature.

Even so, this is not typically an area where we are concerned about speed.

Continuous Integration

Sounds complex but actually very simple:

  • A task runner (simple shell scripts).
  • A near-production environment.
  • A centralised location.
  • A hook to trigger tasks.

Integration needs to be fast!

Continous Integration vs Continuous Deployment vs Continuous Delivery vs Continuous Experimentation

Deployment

  • Triggered on a successful CI build.
  • Choose your deployment strategy.
  • Deploy assets separately?
  • Flush any caches that need it.

Server

Enabling the UX

You should know:

  • Which parts of the application to optimise.
  • How to protect the application - a broken UX is the slowest.

What separates the user and the experience:

  • DNS lookups.
  • A lot of Network Variability.
  • The speed & stability of your application.
  • Assets on the Critical Path

Application Level

What can we cache at this tier?

  • Database queries?
  • Page partials?
  • The entire page?

Application monitoring:

  • Database query timings.
  • Process timings.

Load Balancer or Proxy

Not always necessary, but an opportunity to:

  • Protect struggling servers and distribute load.
  • Report to an auto-scaling mechanism.
  • Run Server Side AB Tests.
  • Handle security threats.

    • And Monitor unhealthy servers.

    Monitor and report on load Balancer latency.

    Useful for alerting because it cuts out variability of the network.

    CDN

    Store static files on edge nodes closer to the user.

    • The entire page.*
    • Static Assets.
    • The DNS lookup.

    * Caveats with page caching: Cookies, Auth, Localisation.

    Uh oh, now everything is cached…

    • Have cache busting mechanisms.
    • Expose what is cached, and where, to the team.

    Browser

    Fetch static assets:

    • Use a Cookieless domain.
    • Cached indefinitely.
    • Versioned e.g. application-e8f29d9aae0aa79.css
    • Store them locally - Browser cache, LocalStorage…

    And kick off client-side monitoring:

    • RUM Timings.
    • Javascript Errors.
    • Custom user events.

    User

    Collect Metrics

    • Measure everything - expose metrics that matter.
    • Collect metrics using tools like statsD, Logstash or even a 1x1.
    • Visualise them - Graphite, DataDog, Kibana.
    • Figure out what they mean!
    • Alert on them.

    Most importantly, expose them to your team…

    Use the Feedback

    Let those metrics help you make your next piece of code, next feature, and ultimately the product, better.
